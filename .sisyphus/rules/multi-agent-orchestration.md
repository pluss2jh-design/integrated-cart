# 멀티 에이전트 오케스트레이션 규칙

이 규칙은 Sisyphus가 다른 에이전트에게 작업을 위임할 때 따라야 하는 지침입니다.

## 에이전트 선택 기준

Sisyphus는 작업의 성격에 따라 아래 기준으로 에이전트를 선택하고 위임합니다:

### 에이전트별 전문 영역

| 에이전트 | 전문 영역 | 언제 위임하는가 |
|----------|-----------|----------------|
| **Oracle** | 아키텍처, 심층 디버깅 | 시스템 설계, 복잡한 버그, 기술 스택 결정 |
| **Prometheus** | 전략 기획, 작업 분해 | 대형 기능 구현 계획, 로드맵 작성 |
| **Librarian** | 문서/코드 리서치 | API 사용법, 라이브러리 선택, 레퍼런스 검색 |
| **Explore** | 코드베이스 탐색 | 파일 구조 파악, 관련 코드 위치 찾기 |
| **Hephaestus** | 코드 작성/수정 | 실제 구현, 리팩토링, 파일 편집 |
| **Metis** | 계획 검토 | 실행 전 계획 타당성 검증 |
| **Momus** | 코드 리뷰 | 완성된 코드의 품질 검토 |
| **Atlas** | 의존성 분석 | 변경의 영향 범위 파악 |
| **Multimodal-Looker** | 이미지 분석 | 스크린샷, 디자인 파일, 다이어그램 해석 |

## 위임 원칙

1. **단일 책임**: 하나의 에이전트에게 하나의 명확한 작업만 위임
2. **컨텍스트 전달**: 위임 시 필요한 컨텍스트를 충분히 제공
3. **순차적 파이프라인**: 이전 에이전트의 결과가 다음 에이전트의 입력이 됨
4. **병렬 실행**: 독립적인 작업은 여러 에이전트가 동시에 처리

## 표준 워크플로우 패턴

### 신규 기능 구현 패턴
```
Sisyphus (계획)
  → Explore (코드베이스 이해)
  → Librarian (필요 라이브러리 리서치)
  → Oracle (구현 아키텍처 설계)
  → Prometheus (구현 단계 세분화)
  → Hephaestus (코드 작성 - 반복)
  → Momus (코드 리뷰)
  → Hephaestus (수정 적용)
```

### 버그 수정 패턴
```
Sisyphus (문제 파악)
  → Explore (버그 위치 확인)
  → Oracle (근본 원인 분석)
  → Hephaestus (수정 적용)
  → Momus (수정 결과 검증)
```

## 응답 형식

각 에이전트는 작업 완료 후 다음 형식으로 Sisyphus에게 보고:
```
[에이전트명] 작업 완료
- 수행한 작업: ...
- 결과: ...
- 다음 추천 작업: ...
- 주의사항: ...
```
